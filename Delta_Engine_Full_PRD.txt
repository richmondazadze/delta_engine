1. Executive Summary
Delta Engine is a cloud-native trade copier SaaS that allows traders to connect one or more master trading accounts
and replicate trades into multiple follower accounts with configurable risk, lot sizing, symbol mapping, execution rules,
and audit tracking.
The platform’s long-term goal is to become a professional-grade copying infrastructure layer for retail traders,
prop-firm traders, account managers, signal providers, and trading communities.
Unlike traditional local trade copiers that require users to install Expert Advisors, rent VPS machines, or keep personal
trading terminals online, Delta Engine will provide a managed cloud environment where the system handles terminal
uptime, connection monitoring, execution routing, risk controls, and account-level protection.
The first build will focus on MT5-to-MT5 cloud-hosted copying using Python-controlled MetaTrader 5
terminal workers.
2. Product Vision
2.1 Vision Statement
Delta Engine will be the control center for serious multi-account traders. A trader should be able to log in, connect
accounts, choose one account as the master, define follower rules, set risk limits, and let Delta Engine handle copying
with transparency, safety, and speed.
2.2 Product Philosophy
 Reliability before speed: A fast copier that misfires is useless. The first goal is accurate copying, clean state
tracking, and safe execution.
 Risk-first architecture: The system must prevent bad trades, over-sizing, over-exposure, and prop-firm rule
violations before they happen.
 Cloud convenience: Users should not need to manage their own VPS or keep a laptop running.
 Audit everything: Every copied trade, failed trade, skipped trade, SL/TP modification, risk lockout, and broker error
should be logged.
 Build narrow first, expand later: MT5 first. Then MT4 and other platforms after the core engine is proven.

5.3 Core Components
Next.js Dashboard
The dashboard is the user control surface. Users can sign up, log in, connect trading accounts, view account status,
create copier relations, configure lot sizing, configure risk rules, start/stop copying, view copied trades, inspect errors,
and view account performance.
Supabase Layer
Supabase handles authentication, user profiles, PostgreSQL database, Row-Level Security, realtime updates, storage
for non-sensitive files, and admin-side data access.
FastAPI Control API
FastAPI handles JWT validation, account creation, credential encryption, copier relation management, worker
commands, risk configuration, execution log APIs, admin APIs, and future public APIs.
Worker Orchestrator
The orchestrator assigns account sessions to available workers. It starts and stops terminal workers, monitors health,
detects stale sessions, schedules reconnects, and manages per-user/account isolation.
Managed MT5 Terminal Workers
Each worker controls one or more MetaTrader 5 terminal sessions. Workers initialize MT5, authenticate trading
accounts, fetch account info, fetch positions/orders, detect trade changes, send follower orders, update SL/TP, close
follower positions, return broker error codes, and report health status.
6. Core Technical Strategy

MT5 is the best first platform because Python-side experimentation is more realistic than MT4. The first technical
milestone is not the dashboard. The first milestone is to connect Python to MT5, read account state, read open
positions, place a demo trade, and close a demo trade.

8. Functional Requirements
8.1 Authentication
 FR-AUTH-1: Users must be able to register with email and password.
 FR-AUTH-2: Users must be able to log in and log out.
 FR-AUTH-3: Users must be able to reset password.
 FR-AUTH-4: Frontend requests to FastAPI must include a Supabase JWT.
 FR-AUTH-5: FastAPI must validate JWTs before allowing access to protected routes.
8.2 Trading Account Management
 FR-ACC-1: Users must be able to add a trading account with platform type, login, broker server, password, label, and
account role preference.
 FR-ACC-2: The system must encrypt broker passwords before storing them.
 FR-ACC-3: The frontend must never receive a stored password after account creation.
 FR-ACC-4: Users must be able to delete a trading account.
 FR-ACC-5: Deleting an account must disable copier relations using that account.
 FR-ACC-6: The system must show connection status: connected, disconnected, auth failed, terminal unavailable,
broker unavailable, disabled, or locked.
8.3 Copier Relations
 FR-COPY-1: Users must be able to create a master-to-follower copier relation.

8.5 Lot Sizing
 FR-LOT-1: Fixed lot mode must execute the same configured lot size regardless of master trade size.
 FR-LOT-2: Multiplier mode must scale the follower lot based on master lot.
 FR-LOT-3: The system must normalize lot size based on broker constraints: minimum lot, maximum lot, lot step, and
margin availability.
 FR-LOT-4: If calculated lot violates broker requirements, the system must round safely, reject the trade, or use
configured fallback behavior.
8.6 Symbol Mapping
 FR-SYM-1: The system must support manual symbol mapping.
 FR-SYM-2: The system must support automatic suffix/prefix detection in the future.
 FR-SYM-3: The system must store per-broker symbol metadata including lot step, digits, point size, contract size,
spread, and session status.
8.7 Risk Engine
 FR-RISK-1: Max Daily Loss - disable copying, optionally close exposure, log reason, notify user.
 FR-RISK-2: Max Total Loss - allow users to set a maximum total account loss.

 FR-RISK-3: Max Lot Per Trade - cap the maximum copied lot size.
 FR-RISK-4: Max Open Positions - limit total concurrent positions per account.
 FR-RISK-5: Symbol Whitelist - allow only selected symbols.

 FR-RISK-10: Auto-Flatten - optionally close all exposure when a risk breach occurs.
 FR-RISK-11: News Pause - future feature to pause copying before and after major news.
8.8 Execution Engine
 FR-EXEC-1: Send buy/sell market orders to follower accounts.
 FR-EXEC-2: Include SL/TP during order placement where possible.
 FR-EXEC-3: If broker rejects SL/TP during initial placement, retry SL/TP modification after entry.
 FR-EXEC-4: Capture broker return codes.
 FR-EXEC-5: Classify execution result: success, rejected, partial, skipped by risk, skipped by slippage, failed auth,
failed connection, insufficient margin, symbol unavailable, duplicate ignored.
 FR-EXEC-6: Never endlessly retry a failed trade.
 FR-EXEC-7: Retry count must be configurable; MVP default max retries = 1.
8.9 Execution Logs
 FR-LOG-1: Every trade signal must create an execution event.
 FR-LOG-2: Every follower attempt must be logged separately.
 FR-LOG-3: Logs must include master ticket, follower ticket, symbol, side, lots, prices, slippage, latency, status, error
message, broker return code, and timestamp.
 FR-LOG-4: Users must be able to filter logs by account, copier relation, symbol, status, and date range.

16. Main Technical Risks
16.1 MetaTrader Terminal Dependency
Delta Engine depends on trading terminals, not clean universal broker APIs. This creates challenges around terminal
installation, terminal login, broker server selection, terminal crashes, Windows/Wine compatibility, multiple terminals
per machine, and account isolation.
16.2 Credential Trust
Cloud-native architecture may require holding encrypted broker credentials. This is powerful but sensitive. Security
must be taken seriously from day one.
16.3 Broker Differences
Different brokers have different symbols, suffixes, lot steps, minimum lots, spreads, stop-level rules, execution modes,
and margin rules.
16.4 Position Model Differences
MT5 can be hedging or netting. This can affect copying behavior significantly.
16.5 Latency Claims
Delta Engine should not market ultra-low latency early. Early positioning should focus on reliable copying, clear logs,
risk protection, zero-install setup, and prop-firm-friendly controls.
